# Clinic Desktop App Planning

## Description

A cross-platform desktop application designed for small clinics to efficiently manage patient records, medical history, prescriptions, and clinic medicine inventory. The app supports role-based access control, real-time notifications, comprehensive audit logging, and integrates with external messaging services for critical system alerts. Built for reliability and extensibility by leveraging modern web and desktop technologies.

---

## Tech Stack

- **Electron JS**: Desktop application framework for building cross-platform apps with web technologies.
- **Java**: Progressive Node.js backend framework for scalable and maintainable server-side applications.
- **MySQL**: Relational database for storing clinic data, patient records, inventory, etc.
- **Docker**: Containerization for consistent development, deployment, and orchestration.
- **GitHub Actions**: CI/CD pipeline for automated testing and deployment.
- **Slack/Telegram Bot Integration**: Extensible notification service for critical system alerts.
- **Modern Frontend Stack**: (e.g., React, TypeScript, or similar) for UI (to be finalized as needed).
- **Others**: Additional libraries as needed for authentication, notifications, etc.

## 5. Build Config & Docker Compose Structure

- All build and deployment configuration files are placed under a top-level `build-config/` directory.
- Config files are grouped into two subdirectories: `dev/` for development and `prod/` for production.
- The development Docker Compose setup includes containers for:
  - The main backend service (Java, JEE)
  - The frontend (Electron/React, if dockerized)
  - Any supporting services (e.g., email mock, Redis, etc.)
- The production Docker Compose is optimized for deployment, using a single MySQL (or managed DB), production environment variables, and any production-only settings.

### Example Directory Structure

```
build-config/
├── dev/
│   ├── docker-compose.dev.yml           # Compose for local development, includes master-slave DB
│   ├── backend.Dockerfile               # Backend (NestJS) Dockerfile for dev
│   └── .env.example                     # Example environment variables for dev
├── prod/
│   ├── docker-compose.prod.yml          # Compose for production deployment
│   ├── backend.Dockerfile               # Backend (NestJS) Dockerfile for prod
│   └── .env.example                     # Example environment variables for prod
```

## 6. Database Schema (DBML for dbdiagram.io)

```dbml
Table users {
  id serial [pk]
  username varchar [unique, not null]
  password_hash varchar [not null]
  full_name varchar
  email varchar
  is_active boolean [default: true]
  is_verified boolean [default: false]      // For email verification
  created_at timestamp [default: `now()`]
  updated_at timestamp
}

Table roles {
  id serial [pk]
  name varchar [unique, not null]
  description varchar
}

Table permissions {
  id serial [pk]
  name varchar [unique, not null]
  description varchar
}

Table user_roles {
  id serial [pk]
  user_id int [ref: > users.id]
  role_id int [ref: > roles.id]
}

Table role_permissions {
  id serial [pk]
  role_id int [ref: > roles.id]
  permission_id int [ref: > permissions.id]
}

Table patients {
  id serial [pk]
  full_name varchar [not null]
  gender varchar
  date_of_birth date
  phone varchar
  address varchar
  medical_history text
  examination_date date
  vital_signs jsonb
  examination_notes text
  diagnosis varchar
  created_at timestamp [default: `now()`]
  updated_at timestamp
}

Table medicines {
  id serial [pk]
  name varchar [not null]
  description varchar
  unit varchar
  price decimal(10,2)
  manufacturer varchar
  batch_number varchar
  expiry_date date
  created_at timestamp [default: `now()`]
  updated_at timestamp
}

Table medicine_inventory {
  id serial [pk]
  medicine_id int [ref: > medicines.id]
  quantity int [not null]
  location varchar
  last_updated timestamp [default: `now()`]
}

Table medicine_transactions {
  id serial [pk]
  medicine_id int [ref: > medicines.id]
  type varchar [not null]             // INBOUND, OUTBOUND, ADJUSTMENT
  quantity int [not null]
  transaction_date timestamp [default: `now()`]
  note varchar
  related_prescription_id int [ref: > prescriptions.id]
  created_by int [ref: > users.id]
}

Table prescriptions {
  id serial [pk]
  patient_id int [ref: > patients.id]
  created_by int [ref: > users.id]
  created_at timestamp [default: `now()`]
}

Table prescription_medicines {
  id serial [pk]
  prescription_id int [ref: > prescriptions.id]
  medicine_id int [ref: > medicines.id]
  quantity int
  dosage varchar
  note varchar
}

Table allowed_ips {
  id serial [pk]
  ip_address varchar [unique, not null]
  description varchar
  created_by int [ref: > users.id]
  created_at timestamp [default: `now()`]
}

Table audit_logs {
  id serial [pk]
  user_id int [ref: > users.id]
  action varchar
  detail text
  created_at timestamp [default: `now()`]
}

Table system_logs {
  id serial [pk]
  log_level varchar [not null]   // e.g., INFO, WARNING, ERROR, CRITICAL
  message text [not null]
  meta jsonb
  created_at timestamp [default: `now()`]
}

Table notifications {
  id serial [pk]
  user_id int [ref: > users.id]
  message text
  type varchar                   // e.g., success, warning, error, seller_alert
  read boolean [default: false]
  created_at timestamp [default: `now()`]
}
```

## 7. Function Details

### FUNC-01: User Login

Authenticate users with email and password, then require OTP sent via email. MFA is enforced for all users using the better-auth library.

### FUNC-02: User Logout

Invalidate user session and log the action.

### FUNC-03: Create User

Allow admin/manager to create a new user by email. Sends password setup email; user must set password and verify before login.

### FUNC-04: Edit User

Edit user information (except for their roles).

### FUNC-05: Deactivate/Activate User

Enable/disable user accounts.

### FUNC-06: Assign Role to User

Assign system roles to a user.

### FUNC-07: Create/Edit Role

Create or modify system roles.

### FUNC-08: Assign Permissions to Role

Assign or modify permissions for a role.

### FUNC-09: Edit Allowed IPs

Manage list of allowed IP addresses for system access.

### FUNC-10: Create Patient

Register new patient, including personal info, medical history, and examination fields.

### FUNC-11: View Patient Detail

View comprehensive patient details, including medical history and past examinations.

### FUNC-12: Search Patient

Search for patients by name, phone, or other attributes.

### FUNC-13: Create/Edit Medicine

Add new medicines or edit details of existing ones.

### FUNC-14: View Medicine List

See all available medicines and their details.

### FUNC-15: Create Prescription

Create a new prescription for a patient, including medicine list and dosage.

### FUNC-16: View Prescription History

View all past prescriptions for a given patient.

### FUNC-17: Change Password

User can change their own password.

### FUNC-18: Audit Log

View actions history for compliance and monitoring.

### FUNC-19: Seller Prescription Notification

Notify sellers (via in-app/Electron Notification) when a prescription is created.

### FUNC-20: In-app and System Notifications

Display system/user notifications to users in-app, leveraging Electron Notification API.

### FUNC-21: Critical System Error Notification

Send critical error logs to Slack or Telegram bot for system monitoring.

### FUNC-22: Create Medicine Inventory Record

Add a new inventory record for a medicine (new stock, received batch, etc.).

### FUNC-23: Update Medicine Inventory Record

Update inventory record for a medicine (adjustment, correction, etc.).

### FUNC-24: Delete Medicine Inventory Record

Delete a medicine inventory record as needed.

### FUNC-25: View Medicine Inventory

View all inventory records for medicines, including stock, batch, and expiry details.

### FUNC-26: Password Setup (by invite link)

Allows a new user (created by admin/manager) to set their password via a secure link sent to their email. Activates the user account and marks it as verified upon successful password creation. Handles expired/invalid links securely.

#### Log & Notification System

- **system_logs**: Logs system events/errors (with severity).
- **notifications**: Records user-facing notifications (success, info, warning, error, seller alerts).
- **Notification Service**: Handles notifications; by default will notify sellers and send critical errors to a Slack bot (can be swapped for Telegram, etc.).
- **Electron Notification API**: Used for displaying native OS notifications for users (including sellers).

#### Authentication & Account Management Notes

- The backend authentication will use the `better-auth` library within the NestJS framework.
- Multi-Factor Authentication (MFA) is required for all users. Each login requires email, password, and a One-Time Password (OTP) sent to the user's registered email address.
- OTPs are generated and sent upon each login attempt. They must expire after a short period and are validated on login.
- When an admin or manager creates a new user account, the system sends a password setup email to the new user's registered email.
- New users must set their password and verify their email address before they can log in.
- All authentication, OTP, and verification events are logged in the audit log.
- Security measures include OTP expiration, rate limiting, and secure handling of sensitive data.

## 8. Notes

- All screens and API endpoints enforce authorization based on roles/permissions.
- Employees have restricted patient data viewing (hide sensitive info as needed).
- Medicine inventory CRUD is controlled by dedicated permissions for separation of duties.
- Audit log tracks all important actions for compliance.
- Electron’s Notification API will be used for native or in-app notifications.
- Sellers are notified immediately (via in-app and/or native notifications) when a prescription is created.
